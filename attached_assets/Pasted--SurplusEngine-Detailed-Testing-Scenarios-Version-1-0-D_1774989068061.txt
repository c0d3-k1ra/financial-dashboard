## 1. Accounts

### 1.1 Account Creation

| # | Scenario | Steps | Expected Result |
|---|----------|-------|-----------------|
| A-01 | Create a Bank account with valid data | POST `/api/accounts` with `{ name: "HDFC Savings", type: "Bank", currentBalance: 50000 }` | 201 Created. Account appears in list with balance ₹50,000.00. |
| A-02 | Create a Credit Card account with all optional fields | POST with `{ name: "ICICI Amazon", type: "Credit Card", currentBalance: 12500, creditLimit: 200000, billingDueDay: 15 }` | 201 Created. `creditLimit` and `billingDueDay` persisted. |
| A-03 | Reject duplicate account name | Create two accounts with `name: "SBI"` | Second request returns 409/422 conflict (unique constraint on `name`). |
| A-04 | Reject billingDueDay outside 1-31 | POST with `billingDueDay: 0` and again with `billingDueDay: 32` | 422 Validation Error for both. |
| A-05 | Create Bank account — creditLimit and billingDueDay ignored | POST Bank account with `creditLimit: 100000` | Account created; `creditLimit` and `billingDueDay` are null. |
| A-06 | Create account with zero balance | POST with `currentBalance: 0` | 201 Created with balance ₹0.00 (default). |

### 1.2 Account Update

| # | Scenario | Steps | Expected Result |
|---|----------|-------|-----------------|
| A-07 | Update account name | PUT `/api/accounts/:id` with new name | Name updated, no balance side-effects. |
| A-08 | Update CC credit limit | PUT with `creditLimit: 300000` | Credit limit updated; outstanding balance unchanged. |
| A-09 | Update billingDueDay to boundary value (31) | PUT with `billingDueDay: 31` | Stored as 31. CC due-date logic clamps to actual month length in Feb. |

### 1.3 Account Deletion

| # | Scenario | Steps | Expected Result |
|---|----------|-------|-----------------|
| A-10 | Delete account with zero transactions | DELETE `/api/accounts/:id` where no txns exist | 200 OK. Account removed from list. |
| A-11 | Delete account with linked transactions | DELETE account that has at least one transaction | 409/422 rejection — "Cannot delete account with existing transactions." |
| A-12 | Delete account referenced as `toAccountId` in a transfer | Same as above but for the destination side of a transfer | Deletion blocked. |

### 1.4 Account Reconciliation

| # | Scenario | Steps | Expected Result |
|---|----------|-------|-----------------|
| A-13 | Reconcile with higher actual balance | Account tracked: ₹45,000. Reconcile to ₹50,000. | Adjustment transaction of +₹5,000 created. Account balance now ₹50,000. |
| A-14 | Reconcile with lower actual balance | Account tracked: ₹50,000. Reconcile to ₹47,500. | Adjustment transaction of −₹2,500 created. Balance now ₹47,500. |
| A-15 | Reconcile with no difference | Reconcile to current tracked balance | Either no adjustment transaction created, or a ₹0 adjustment — verify consistent behavior. |

## 13. Edge Cases & Error Handling

| # | Scenario | Steps | Expected Result |
|---|----------|-------|-----------------|
| E-01 | Empty state — no accounts | New user, no accounts created | Dashboard gracefully shows zero balances, empty charts, prompt to add an account. |
| E-02 | Empty state — no transactions | Accounts exist, no transactions logged | Dashboard shows account balances, ₹0 income/expense/surplus, empty trend charts. |
| E-03 | Empty state — no goals | No goals created | Goal Vault page shows empty state message / CTA to create first goal. |
| E-04 | Decimal precision | Create transaction with ₹99.999 | Rounded or truncated to ₹100.00 (2 decimal places per `numeric(12,2)`). |
| E-05 | Concurrent balance updates | Two transactions created simultaneously for the same account | Both balance adjustments applied correctly; no race condition losing one update. |
| E-06 | API returns 500 | Simulate backend error | Frontend displays error toast/message. Does not crash. React Query retries if configured. |
| E-07 | Network offline | Disconnect network, attempt action | Frontend shows connection error. No silent data loss. |
| E-08 | Very long description | Transaction with 1,000-character description | Either accepted and truncated in display, or rejected with validation error. |
| E-09 | Special characters in account/category names | Name: `O'Brien & Sons "Special" <Account>` | Properly escaped in DB and UI. No XSS or SQL injection. |
| E-10 | Monthly config — starting balance | Set `startingBalance: 100000` for 2026-04 | Dashboard summary for April incorporates starting balance. |
| E-11 | Surplus ledger uniqueness | Attempt two surplus ledger entries for same month | Second insert fails (unique constraint on `month`). |
| E-12 | Goal vault uniqueness | Create two vaults with the same name | Second insert fails (unique constraint on `name`). |

## 16. Cross-Cutting Concerns

### 16.1 Data Integrity

| # | Scenario | Expected |
|---|----------|----------|
| X-01 | Sum of all account adjustments from transactions equals current account balances | Balances are always derivable from transaction history + initial balance. |
| X-02 | Surplus allocations total never exceeds calculated surplus for a month | Allocation endpoint validates against available surplus. |
| X-03 | Goal `currentAmount` matches sum of its surplus allocations + direct updates | Consistency between goals and surplus_allocations tables. |
